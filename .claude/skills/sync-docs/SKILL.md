# sync-docs — 项目知识库同步

开发完成后手动执行，分析代码变更并更新 `docs/` 目录中的项目知识库文档。

## 使用场景

- 开发完成后，执行 `/sync-docs` 更新知识库
- 用户说"更新文档"、"同步知识库"、"sync docs"

## 执行步骤

### 1. 获取变更范围

```bash
# 默认分析最近一次提交的变更
git diff --name-only HEAD~1..HEAD

# 如果用户指定了范围，用 git log 确认
git log --oneline <start>..HEAD
git diff --name-only <start>..HEAD
```

### 2. 确定需要更新的文档

根据以下映射关系判断：

| 变更文件/目录 | 需要更新的文档 |
|--------------|--------------|
| README.md | 01-项目概览.md、02-架构设计.md、06-API参考手册.md |
| frontend/ | 03-前端开发指南.md |
| backend-ts/ | 04-TS后端开发指南.md |
| backend-py/ | 05-Python后端开发指南.md |
| GUIDE.md | 08-初学者教程.md |
| DEVELOPMENT.md | 07-开发环境与调试.md |
| .github/workflows/ | 07-开发环境与调试.md |
| package.json（根目录） | 01-项目概览.md、07-开发环境与调试.md |
| vitest.config.ts / 测试文件 | 07-开发环境与调试.md + 对应开发指南 |

### 3. 分析变更内容

对每个受影响的文档：
1. 读取变更的源文件（用 `git diff` 或直接 Read）
2. 读取当前 `docs/` 文档内容
3. 判断需要追加新章节还是更新已有章节

### 4. 更新文档

- **新增功能**：在对应文档中追加新章节
- **修改已有功能**：找到对应章节，用 Edit 工具更新
- **保留原文档结构**：不删除或重写未受影响的部分

### 5. 更新变更记录

在 `docs/09-变更记录.md` 末尾追加变更摘要：

```markdown
## vX.X.X — YYYY-MM-DD
### 简要描述
- 变更项 1
- 变更项 2
```

### 6. 报告更新结果

告知用户更新了哪些文档、做了什么变更。

## 注意事项

- 只更新受变更影响的文档，不要修改无关文档
- 保留原文档的 Markdown 结构和格式
- 如果变更很小（如 typo 修复），可以跳过文档更新
- 新增的测试文件只在引入了新测试框架时才需要更新文档
